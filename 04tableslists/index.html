Topic

List-like Views

List-like Views

  • TableView
  • ListView
  • ScrollView
Your options for list-like views include the three shown here. Titanium does not offer a grid-like view, because the native platforms offer no such controls. Your best bet for a grid-control would be nested ScrollViews.

TableView Characteristics

  • Vertically scrolling container of data presented as rows
  • Tables can contain sections, which contain rows, which can contain views
  • There's an implicit section if you don't add one
  • Basic row contents: title, leftImage, rightImage, indicators
  • Optional: add Views as children
  • Event listeners on table (preferred), section, rows, or child views
TableViews are not a grid component, rather a list-like vertically scrolling container of rows. It is the traditional Titanium UI component for list-like views. Works across platforms, but performance can be slow with large or visually complex tables.

ListView Characteristics

  • Vertically scrolling container of data presented as items
  • ListViews contain sections, which contain ListItems, which render ListDataItems
  • ListSection is required; add ListItems to sections not the list itself
  • ListItem = ListDataItem + ItemTemplate
  • Event listeners on list itself or on child components within the ListItems
ListViews were introduced with Titanium 3.0 as a replacement for TableViews. They feature improved rendering and scrolling speed at a cost of more complex programming.

ScrollView Characteristics

  • Vertically or horizontally scrolling container of child views
  • ScrollViews contain child Views of nearly any type
  • Don't put TableViews or ListViews within a ScrollView
  • Put event listeners on the child views
Some developers use ScrollViews to create custom scrolling lists. In some situations, ScrollViews offer better performance than TableViews. However, eventing is more difficult to rig up and can slow performance in real-world use.

Why Use One over the Other?

  • TableViews - simpler API and usage
  • TableViews - direct manipulation of child views
  • TableViews - lots of code examples in docs and sample apps
  • ScrollViews - horizontal scrolling
  • ListViews - speed!   Faster rendering and smoother scrolling even with large data sets

TableViews

TableView Basics

var table = Ti.UI.createTableView();

We've used tables throughout the TCD and TCE course. So, we won't spend a lot of time on how to use them. Instead, we'll focus on a few things you can do with TableViews that you can't do as easily with ListViews.

Headers and Footer Views


$.table.header = 'TableView examples and test cases';
$.table.headerView = $.simpleTableAPIView;
$.table.footerView = $.anotherView;
You can use the headerView and footerView properties to place Ti.UI.View objects as headers and footers, and those you can style as needed.

Table Search


var searchbar = Ti.UI.createSearchBar({
   barColor:'#385292',
   showCancel:false
});

$.table.search = searchbar;

You can enable a search bar on the TableView, which performs a wildcard text search on the row titles (not child views, such as labels). Rendering varies by platform, as do some of the features supported.

Accessing Child Views


<TableView id="table">
  <TableViewRow title="row">
    <Label text="A label"></Label>
    <ImageView image="/someimage.png"/>
  </TableViewRow>
</TableView>
					

$.table.addEventListener('click', function(e){
  // e.row.children[1] == the ImageView
})
Within an event listener, you can access child Views of a TableViewRow by accessing the children[] array.

Row Indicators


/* in the TSS */
"#someRow": {
  hasChild: true,
  hasDetail: true,
  hasCheck: true
}
Per Apple's HIG, these icons imply the following meaning:
  • hasChild — indicates sub-table or additional rows
  • hasDetail — indicates a detail view or alert will appear when row is tapped (not supported on Android)
  • hasCheck — on/off or yes/no indicator

Table View Performance

  • className allows Titanium to reuse native objects
  • setData() or table.data faster than append()
  • Load only as much data as needed
  • If your table has 1,000s of custom rows, you might need to rethink your UX design
  • Don't use child Views, just title & row properties


<TableView id="table">
  <TableViewRow className="anystring"/>
</TableView>
  • Adding className enables Ti & underlying OS to reuse table row objects and optimize memory
  • A setData() call for a lot of rows is faster than repeated appendRow() calls
  • Perhaps you can use alternate UI paradigm to create smaller tables
  • Dawson's Lazy Loaded Tables http://j.mp/rbL32h

Topic

ListViews
ListViews are vertically-scrollable containers of data presented as items. They are a data-oriented component. You don't have direct access to child Views within a ListItem.

ListView Sample (Traditional)


var myTemplate = {
  childTemplates: [{
      type: 'Ti.UI.ImageView',
      bindId: 'pic',
      properties: { width: '50dp', height: '50dp', left: 0 }
    }, {
      type: 'Ti.UI.Label',
      bindId: 'info',
      properties: {
        color: 'black',
        font: { fontFamily:'Arial', fontSize: '20dp', fontWeight:'bold' },
        left: '60dp', top: 0,
      }
    }]
};

var listView = Ti.UI.createListView({
  templates: { 'template': myTemplate }, defaultItemTemplate: 'template'
});

var sections = [];
var fruitSection = Ti.UI.createListSection({ headerTitle: 'Fruits'});
var fruitDataSet = [
  { info: {text: 'Apple'}, pic: {image: '/apple.png'}},
  { info: {text: 'Banana'}, pic: {image: '/banana.png'}}
];
fruitSection.setItems(fruitDataSet);
sections.push(fruitSection);
listView.setSections(sections);
Explain the code: we define templates for our items, then we create the list view and assign the templates. Next we create sections and add items to it (you can't add items directly to a ListView). Finally, we add the sections to the list view.

ListView Sample (Alloy)


<ListView id="listView" defaultItemTemplate="template">
  <Templates>
    <ItemTemplate name="template">
      <ImageView bindId="pic" id="icon" />
      <Label bindId="info" id="title" />
    </ItemTemplate>
  </Templates>

  <ListSection headerTitle="Fruit">
    <ListItem info:text="Apple" pic:image="/apple.png" />
    <ListItem info:text="Banana" pic:image="/banana.png" />
  </ListSection>
</ListView>
					

// in associated TSS file:
"#icon" : {
  width: '50dp', height: '50dp', left: 0
},
"#title" : {
  color: 'black',
  font: { fontFamily:'Arial', fontSize: '20dp', fontWeight:'bold' },
  left: '60dp', top: 0
}
The Alloy code is perhaps simpler: again, we define our templates, assign them to the list view, define our section, and its items. We can put styling into the TSS file with this technique.

ItemTemplates

  • You must define a template for the ListView
  • A ListView can have multiple templates, but one must be the defaultTemplate
  • Each template can have child templates, which define what components are contained in the ListItem (row)
  • There are built-in templates supported on iOS
See the API docs and guides for info on the built-in templates on iOS.

Item Accessories

Assign to template or to the items directly (e.g. in an event listener)

ListView Eventing


// use the special 'itemclick' event with ListViews
listView.addEventListener('itemclick', function(e){
  // grab a handle to the list item (row) that was clicked
  var item = e.section.getItemAt(e.itemIndex);
  // modify properties of list item:
  if (item.properties.accessoryType == Ti.UI.LIST_ACCESSORY_TYPE_NONE) {
      item.properties.accessoryType = Ti.UI.LIST_ACCESSORY_TYPE_CHECKMARK;
  } else {
      item.properties.accessoryType = Ti.UI.LIST_ACCESSORY_TYPE_NONE;
  }

  // once you've modified the properties, you must update the item
  e.section.updateItemAt(e.itemIndex, item);  

  // use bindId (not bindID!) to determine which child component was clicked
});
ListView events use the special itemclick event. You cannot access tiems directly from the event object as you can with TableViews. Instead, use the section's getItemAt() method. In this example, we're modifying the properties of the item. Note that we have to call updateItemAt() when we're done. The e.bindId will correspond to the ID you set in the template and will be undefined if the user doesn't click on a child component.

Eventing, part two


var someTemplate = {
  childTemplates: [
    {
      type: 'Ti.UI.Label', // Use a label
      bindId: 'rowtitle',  // Bind ID for this label
      properties: {        // Sets the Label.left property
          left: '10dp'
      }
    },
    {
      type: 'Ti.UI.Button',   // Use a button
      bindId: 'button',       // Bind ID for this button
      properties: {           // Sets several button properties
          width: '80dp',
          height: '30dp',                        	
          right: '10dp',
          title: 'press me'
      },
      events: { click : report }  // Binds a callback to the button's click event
    }
  ]
};

function report(e) {
  Ti.API.info(e.type);
}
In this case, we're attaching an event listener directly to a child element (a button) and associating it with a named function defined elsewhere. It would be a poor idea to define an anonymous function in line, as a seprate function would be defined for each item in your list view.

Compare & Contrast

TableView vs. ListView

TableViewListView
View-oriented architecture for maximum flexibility Data-oriented architecture for maximum performance
Sections implicit & can generally be ignored Sections required
Header/footer views, search bars, more Header/footer (string) titles
Bind events to table, sections, rows, or child Views Bind events to ListView or child components (via template)
Relatively easier to code Faster
